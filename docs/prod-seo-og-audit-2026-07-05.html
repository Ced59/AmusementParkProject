Audit production SEO, Open Graph et baisse d'impressions Google

Audit réalisé le 5 juillet 2026 après les déploiements 3.3.16 à 3.3.20, avec tests depuis le PC local et depuis le VPS en SSH root.

Production: amusement-parks.fun Version active: 3.3.20 Commit prod: 04d72ddf VPS: root@srv486819 Date: 2026-07-05

Conclusion

La baisse d'impressions Google ne vient pas principalement du déplacement des sitemaps XML à la racine. Les sitemaps racine sont en 200, les anciens chemins /sitemaps/*.xml redirigent en 308 permanent, et robots.txt pointe bien vers https://amusement-parks.fun/sitemap.xml.

La cause la plus probable identifiée est le cache SSR obsolète du plan de site HTML public : Googlebot recevait une version très pauvre de /fr/sitemap au lieu de la version complète avec les liens profonds. Cette cause a été corrigée par la PR #360, puis sécurisée par la PR #361 qui empêche un ancien run de redéployer une version obsolète après une version plus récente.

Cause Racine

Ce qui n'est pas la cause principale

  • /sitemap.xml répond en 200 et liste 104 sitemaps enfants, tous à la racine du domaine.
  • Les anciens chemins testés /sitemaps/sitemap.xml, /sitemaps/history-articles-fr.xml et /sitemaps/parks-fr.xml répondent en 308 Permanent Redirect vers leur équivalent racine.
  • Google peut suivre ces redirections permanentes. Il peut y avoir un délai de recrawl, mais ce point ne suffit pas à expliquer une chute forte si les pages restent découvertes autrement.

Ce qui explique la chute

  • Avant correction, /fr/sitemap avait servi un cache SSR ancien et incomplet à Googlebot : environ 67 Ko et seulement quelques dizaines de liens, au lieu du rendu complet.
  • La page HTML de plan de site est une source d'interliens importante. Si Googlebot reçoit une version sans liens profonds, il perd un signal de découverte interne pour les parcs, items, timelines et articles.
  • Un ancien déploiement production a ensuite pu repasser après un déploiement plus récent, ce qui a remis la prod sur une version antérieure et entretenu le symptôme. La PR #361 bloque maintenant ce scénario.

État après correction

  • /fr/sitemap sert maintenant 5 702 145 octets et 8 275 liens HTML SSR.
  • Les liens profonds Dennlys Parc, Le Nitro, timeline Nitro et article Nitro sont présents dans le HTML SSR.
  • Les sitemaps XML contiennent la timeline Nitro dans les fichiers history-*.xml et l'article Nitro dans les fichiers history-articles-*.xml pour les 8 langues.

Problèmes Restants

  1. Pages d'entités manquantes : signaux SEO contradictoires.
    Exemple testé : /fr/park/00000000-0000-0000-0000-000000000000/parc-manquant répond bien en HTTP 404, mais le HTML SSR contient meta robots=index,follow, une canonical vers l'URL manquante et des OG génériques. Google devrait respecter le 404, mais il vaut mieux émettre explicitement noindex,follow et éviter une canonical indexable sur une entité absente.
  2. Image OG : dimensions annoncées trop optimistes.
    L'image Nitro b50fc41... est servie en 586x390 malgré ?width=1200 et la meta og:image:width=1200. Ce n'est pas bloquant pour l'indexation, mais ça peut réduire la qualité ou la cohérence des previews sociales.
  3. Images OG génériques sur plusieurs pages publiques.
    Home, liste des parcs, plan du site, dossiers techniques et la fiche Dennlys Parc utilisent encore le logo par défaut. Ce n'est pas une cause de baisse Google, mais c'est moins bon pour les partages.
  4. Plan du site HTML volumineux.
    Le rendu complet est nécessaire pour restaurer les liens, mais il pèse 5,7 Mo. Le cache corrige le coût récurrent, avec un premier rendu observé à 13,73 s depuis le PC puis un cache hit à environ 0,30 s. À surveiller sur VPS modeste.
  5. Copie technique FR sans accents.
    La meta description de /fr/technical contient materiaux et systemes. C'est faible priorité, mais contraire à la règle i18n du projet.

Actions Recommandées

  1. Surveiller Search Console pendant le recrawl de /fr/sitemap, /sitemap.xml et des sitemaps history-*.xml / history-articles-*.xml. Le code côté prod est maintenant cohérent ; il faut laisser Google réexplorer les liens.
  2. Créer une petite PR SEO 404 pour que toutes les entités publiques absentes émettent noindex,follow, sans canonical indexable ni OG génériques indexables.
  3. Créer une PR OG images pour utiliser les logos/photos de parc quand disponibles sur les fiches parc et éviter le logo générique sur les pages à fort partage.
  4. Corriger les dimensions OG : soit produire réellement une image 1200 px, soit ne pas annoncer une largeur fixe quand le binaire servi est plus petit.
  5. Corriger la copie FR technique avec accents et diacritiques.

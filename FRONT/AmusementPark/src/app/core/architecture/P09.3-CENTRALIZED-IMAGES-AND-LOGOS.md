# P09.3 — Centralisation des images, logos et fallbacks

## Objectif

Faire converger le front vers un socle unique de rendu d'image, de résolution d'URL et de fallback,
sans casser la sémantique métier existante.

## Règle métier conservée

`ImageCategory` reste le **grand classement** des images.

Exemples :

- `AVATAR`
- `PARK_LOGO`
- `PARK`
- `PARK_ITEM`

Les **tags** constituent une granularité plus fine, utile pour enrichir la recherche, l'éditorialisation
et les usages spécialisés.

Conséquence importante :

- un logo reste une image du domaine `images` ;
- un logo de parc reste catégorisé `PARK_LOGO` ;
- le tag `logo` est un enrichissement complémentaire, pas un remplacement de la catégorie.

## Décisions techniques de la phase

- centraliser la résolution des sources d'image dans `ImagesApiService` ;
- enrichir `app-image-display` pour qu'il puisse afficher un `imageId`, une URL, un chemin API ou une data URI ;
- réutiliser un modèle UI commun pour les images possédées par une entité (`OwnedImageItem`) ;
- faire converger les écrans admin logos/photos et les écrans avatar vers ce socle ;
- continuer à passer par l'API `images` et non par des circuits front dédiés logo.

## Effet attendu

À partir de P09.3, le front ne doit plus multiplier :

- les `<img>` directs pour des images de domaine ;
- les variantes locales de fallback ;
- les petits view-models images/logos redéclarés par écran ;
- les résolutions d'URL codées à la main.
